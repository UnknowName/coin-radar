# 需求说明书

## 1. 引言

### 1.1 目的
本文档旨在详细说明一个自建加密货币盯盘系统的需求，该系统能够监控多个交易所的市场数据，识别潜在的交易信号，并通过钉钉向用户推送高质量的异动警报。

### 1.2 范围
本系统将从币安、OKX 和 Bybit 三个交易所获取公开市场数据，实现四个核心监控模块，并通过信号过滤层处理后，将结果推送到用户的钉钉。

### 1.3 定义、首字母缩写词和缩略语
- **API**: 应用程序编程接口
- **OI**: 未平仓合约量 (Open Interest)
- **CVD**: 累积成交量差 (Cumulative Volume Delta)
- **z-score**: 标准分数，表示数值与均值之间的标准差数
- **perp/spot**: 永续合约与现货

## 2. 总体描述

### 2.1 产品愿景
创建一个完全本地运行、免费、可定制的加密货币盯盘系统，帮助用户在不依赖第三方工具的情况下，及时捕捉市场异动信号。

### 2.2 系统架构
整体架构为：**三个交易所** → **四个监控模块** → **信号过滤** → **钉钉推送**

### 2.3 用户特征
- 对加密货币交易有一定了解
- 不一定具备编程技能
- 希望获得高质量、低噪音的市场信号
- 能够配置基本的开发环境

### 2.4 运行环境
- 本地 Windows 电脑
- Python 3.12 或 3.14
- Claude Code 工具
- 代理服务器（国内用户）
- 钉钉账号

## 3. 功能需求

### 3.1 数据获取模块

| 编号 | 需求描述 |
|------|----------|
| 3.1.1 | 从币安、OKX 和 Bybit 三个交易所获取公开市场数据 |
| 3.1.2 | 支持异步并发请求，提高数据获取效率 |
| 3.1.3 | 实现令牌桶限流机制，避免被交易所 API 限制 |
| 3.1.4 | 集成 tenacity 重试机制，处理网络抖动问题 |
| 3.1.5 | 使用 ccxt 库统一处理多交易所数据 |

### 3.2 监控模块

#### 3.2.1 山寨币异动扫描

**3.2.1.1 六维加权评分系统**

| 维度 | 权重 | 检测指标 |
|------|------|----------|
| 静默吸筹 | 25% | 24 小时价格波动小于 5%，但 5 分钟"成交量/价格变动"突然飙升 |
| CVD 主动买入异常 | 20% | 累积主动买卖差正向偏离 7 日均值 2 倍以上 |
| 持仓量上升 + 价格平稳 | 20% | 4 小时持仓量增长超过 10%，价格变动小于 3% |
| 大户多空背离 | 20% | 大户多空比与全体用户多空比出现背离 |
| 资金费率背离 | 10% | 价格上涨但资金费率为负或反向 |
| 流动性变薄 | 5% | 订单簿深度比 24 小时均值低 30% 以上 |

**3.2.1.2 推送规则**
- 综合得分 ≥ 60 才推送
- 综合得分 ≥ 80 标记为高优先级

**3.2.1.3 冷却机制**
- 同币种 30 分钟内不重复推送

#### 3.2.2 主流币告警

| 编号 | 需求描述 |
|------|----------|
| 3.2.2.1 | 监控 BTC/ETH/SOL 等主流币的剧烈波动 |
| 3.2.2.2 | 及时提醒用户把握加密货币市场整体风向 |

#### 3.2.3 新合约上线检测

| 编号 | 需求描述 |
|------|----------|
| 3.2.3.1 | 检测币安新上线的合约 |
| 3.2.3.2 | 及时推送新合约信息，用于抢跑早期波动 |

#### 3.2.4 永续/现货量比异常

| 编号 | 需求描述 |
|------|----------|
| 3.2.4.1 | 计算永续合约与现货市场的成交量比例 |
| 3.2.4.2 | 使用 z-score 方法识别统计学上的极端偏离：<br>`z = (当前比值 - 7 天均值) / 7 天标准差` |
| 3.2.4.3 | 当 z-score 绝对值 > 3 时推送警报 |

### 3.3 信号过滤模块

| 编号 | 需求描述 |
|------|----------|
| 3.3.1 | **硬阈值过滤**：基线样本数据至少 12 小时 |
| 3.3.2 | **质量分排序**：<br>- 变异系数（标准差/均值）加分（币种本身稳定，异动才有意义）<br>- ratio>10 标记为高投机度，ratio<0 标记为低投机度<br>- 每轮只保留 top10 信号 |
| 3.3.3 | **冷却机制**：同币种 2 小时内不重复推送 |
| 3.3.4 | **去重机制**：跨模块如果同币同时间触发，只推一次 |

### 3.4 钉钉推送模块

| 编号 | 需求描述 |
|------|----------|
| 3.4.1 | 配置钉钉机器人 Webhook URL |
| 3.4.2 | 配置钉钉机器人密钥（如果需要） |
| 3.4.3 | 设计清晰醒目的推送格式，包含以下信息：<br>- 模块标识 + 币种<br>- z-score 评分 + 偏热/偏冷 + 投机度标签<br>- 当前价格/24 小时涨跌<br>- 成交量/1 小时倍数/24 小时倍数<br>- OI（未平仓合约量）<br>- RSI(14) 指标<br>- 阻力位和支撑位<br>- ratio 详情（当前值、均值 ± 标准差）<br>- 样本信息（数量/时长）<br>- Coinglass 链接 |
| 3.4.4 | 支持钉钉消息的@功能，可以指定提醒特定人员 |
| 3.4.5 | 实现消息发送失败重试机制 |

## 4. 非功能需求

### 4.1 性能需求

| 编号 | 需求描述 |
|------|----------|
| 4.1.1 | 500 个币的扫描能在 30 秒内完成 |
| 4.1.2 | 系统每天稳定推送 20-30 条高质量信号 |

### 4.2 安全需求

| 编号 | 需求描述 |
|------|----------|
| 4.2.1 | 系统只读取公开数据，不需要用户 API 密钥 |
| 4.2.2 | 即使代码出现 bug，也不会威胁到用户账户资金安全 |
| 4.2.3 | 钉钉机器人密钥需要加密存储 |

### 4.3 可靠性需求

| 编号 | 需求描述 |
|------|----------|
| 4.3.1 | 系统应能处理网络抖动和 API 临时不可用的情况 |
| 4.3.2 | 实现指数退避重试机制，确保数据获取的稳定性 |
| 4.3.3 | 钉钉消息发送失败时，需要有备用通知机制 |

### 4.4 可维护性需求

| 编号 | 需求描述 |
|------|----------|
| 4.4.1 | 代码结构清晰，便于用户根据自己的交易逻辑调整参数 |
| 4.4.2 | 权重和阈值应易于修改，以适应不同交易风格 |
